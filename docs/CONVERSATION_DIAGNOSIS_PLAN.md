# 对话式诊断演进方案

> 当前版本 `v0.1.1` 的诊断是「规则优先 + LLM 兜底填表」，外测反馈「AI 味重、不智能、不像在跟 AI 聊」。
> 本方案规划向「**LLM 像维修顾问一样对话 + 规则在背后兜底**」演进，分三阶段落地。

## 1. 背景与问题

外测中用户反馈：诊断过程「不智能」「AI 味」「感觉不到在跟 AI 对话」。根因不在模型，而在**架构**——规则全程主导，LLM 只在最后填几个结构化字段。

## 2. 当前诊断现状

### 2.1 当前流程（`DiagnosisService.handle_message`）

```
用户消息
  ↓
① 内容安全审核（腾讯 TMS）
  ↓
② 【规则】分类 classification_service.classify
   → 关键词模板优先，置信度<0.7 才调 LLM 兜底
  ↓
③ 【规则】风险 risk_service.assess
   → 高风险关键词命中 → 强制 S 级
  ↓
④ 【规则】追问 should_ask + next_questions
   → 从硬编码 QUESTIONS 池取固定 3 个问题
  ↓
⑤ 出结果 _build_result
   → RAG 知识库 + 价格库 + LLM 生成结构化字段
     （LLM 只生成 possible_causes / actions / forbidden 几个 JSON 字段）
```

### 2.2 「AI 味」的 5 个根源

| # | 现象 | 根源 |
|---|---|---|
| 1 | 第一步就被规则判定 | ②③④ 全是规则先跑，LLM 只在最后填字段，用户全程感觉不到「在跟 AI 聊」 |
| 2 | 追问很机械 | ④ 是硬编码固定 3 问，不管用户已说什么都问这 3 个 |
| 3 | AI 不会顺着用户说 | 用户答「电热水器」，AI 不会接「好的，电热水器一般先查…」，直接跳结构化结果 |
| 4 | AI 不记得用户 | 无用户画像，每次从零开始，不记得房屋/上次修过什么/处理偏好 |
| 5 | 结果像填表 | LLM 只填 JSON 字段，没有自然语言的「顾问式」表达 |

**一句话**：现在是「规则优先 + LLM 兜底填表」，目标是「LLM 对话主导 + 规则兜底」——架构方向的转变。

## 3. 目标架构：对话式诊断

### 3.1 核心转变

```
现在：规则主导流程，LLM 填结构化字段
目标：LLM 主导对话，规则做安全/关键字段兜底 + 结构化提取
```

### 3.2 三层设计

```
用户消息
  ↓
【对话层 · LLM 主导】
  - 维修顾问 system prompt（自然、共情、会顺着用户说）
  - 注入：用户画像（房屋/设备/历史）+ 本会话上下文 + 安全边界
  - 输出：自然语言回复（追问/澄清/确认），不是固定问题
  ↓
【兜底层 · 规则】（保留，但不抢戏）
  - 安全兜底：高风险关键词 → 强制 S 级 + 安全动作（覆盖 LLM，PRD §8.4 不变）
  - 关键字段兜底：LLM 漏问「是否靠近电源/有无燃气味」→ 规则提示 LLM 补问
  - 出结果触发：LLM 判断信息够 / 规则判断关键字段齐 / 用户说「够了」
  ↓
【结构化层】
  - LLM 对话同时，提取结构化字段（分类/紧急/原因/建议/价格）
  - 落 DiagnosisResult + 自然语言顾问总结
```

### 3.3 用户画像（让 AI「了解每个用户」）

复用已有表 + 历史：

| 数据 | 来源 | 状态 |
|---|---|---|
| 房屋/房间/城市能级 | `houses` / `rooms` | ✅ 已建（P1-5） |
| 设备档案（品牌/年限） | `devices`（PRD §17.7） | ⏳ 待建 |
| 历史诊断/维修 | `diagnosis_results` / `repair_records` | ✅ 已有 |
| 处理偏好 | 从历史反馈推断 | ⏳ 待积累 |

每次诊断**注入用户画像到 LLM context**，AI 就能说「上次你卫生间漏水修好了，这次是同一个位置吗？」

## 4. 分阶段实施

按「投入小、见效快」排序，每阶段独立可上线。

### 阶段 1：LLM 主导追问（去 AI 味，最见效）⭐⭐⭐

- 把硬编码 `question_service` 固定追问，换成 LLM 生成自然语言追问
- 新增 `conversation_service`：每轮把「用户消息 + 历史 + 安全约束」给 LLM，返回自然追问
- 规则只在 LLM **漏问关键字段**时提示（如没问「是否漏电」→ 注入提示）
- `should_ask` 改为 LLM 判断 + 规则保底（轮数上限 ≤3）
- **工作量** 2-3 天 ｜ **效果**：立刻去机械感，用户感觉「在跟人聊」

### 阶段 2：用户画像注入（让 AI 记得你）⭐⭐⭐

- 新增 `user_profile_service`：聚合 houses/rooms/设备/历史 → 「用户摘要」
- system prompt 注入用户摘要，LLM 据此个性化追问/建议
- 前置：设备档案表 `devices`（PRD §17.7，半天建表+迁移）
- **工作量** 2-3 天 ｜ **效果**：AI「认识」用户，复购/复发追踪的基础

### 阶段 3：顾问式自然结果 ⭐⭐

- `_build_result` 增加 `advisor_summary`：LLM 自然语言顾问总结（维修师傅口吻）
- 前端结果页顶部展示顾问总结，结构化字段在下方
- 复用阶段 1 的 LLM 调用，不额外增加成本
- **工作量** 1-2 天 ｜ **效果**：结果有人情味，不只填表

## 5. 风险与取舍

| 风险 | 应对 |
|---|---|
| 成本上升（LLM 每轮调，1 次→3 次） | 模板兜底（低风险常见问题不调 LLM）+ 轮数 ≤3 + 结果复用对话 LLM |
| 延迟变高 | 流式输出（SSE）+ 短 prompt + 缓存 |
| LLM 不稳/幻觉 | 规则兜底覆盖（安全/关键字段）+ schema 约束 + 输出过 TMS |
| 可控性下降 | system prompt 严格边界 + 关键字段清单 + 评测集回归 |

**底线原则**（PRD §8.4 / §18.6）：**高风险规则永远覆盖 LLM**——燃气/漏电/冒烟，LLM 说啥都不算，规则强制 S 级 + 安全动作。不为「自然」牺牲安全。

## 6. 评测：怎么验证「更智能了」

复用 200 条黄金集 + 新指标：

- **追问命中率**：LLM 追问是否覆盖关键字段（对照 PRD §8.2 字段清单）
- **追问自然度**：人工评 50 条（1-5 分，是否像维修顾问）
- **基线不回归**：分类 95.5% / 紧急 95.0% / 高风险召回 98.1% 不降
- **外测反馈**：问卷加一题「这次对话感觉像跟 AI 还是跟维修师傅？」

## 7. 落地顺序

```
阶段 1（LLM 主导追问）→ 外测对比 → 阶段 2（用户画像）→ 阶段 3（顾问式结果）
```

每阶段独立上线 + 评测回归，不一次推翻重写。
