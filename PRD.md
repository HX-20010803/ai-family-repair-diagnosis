## 2. 产品概述

### 2.1 产品背景

家庭用户在遇到漏水、跳闸、空调不制冷、马桶堵塞、门锁故障等问题时，通常会面临几个共同困境：

- 不知道问题类型和严重程度。
- 不知道能不能自己处理，哪些操作存在安全风险。
- 不知道应该找物业、品牌售后、水电工、开锁师傅还是管道疏通服务。
- 不知道维修报价是否合理，容易被信息不透明影响决策。
- 维修过程和费用难以沉淀，复发后无法追溯历史处理记录。

AI 家庭维修诊断助手希望在用户真正联系维修服务前，先提供一次低门槛、结构化、可追溯的初步诊断，帮助用户快速判断“这是什么问题、严不严重、能不能自己处理、要不要找师傅、大概要多少钱、维修前要注意什么”。

### 2.2 产品定位

AI 家庭维修诊断助手不是泛化 AI 客服，也不是纯维修交易平台，而是面向家庭用户的 AI 维修决策助手。MVP 第一版采用移动端 H5 快速上线，用户通过手机浏览器或微信内置浏览器打开即可完成诊断；后续再适配微信小程序体验版和正式版。中长期优先接入物业、房东和维修服务生态，形成“诊断 -> 报修/维修 -> 记录 -> 复发追踪”的闭环。

产品通过文字、语音、图片等输入方式理解用户描述，结合多轮追问、维修知识库、风险规则、内容安全审核和价格参考库，输出结构化诊断结果，包括故障类型、紧急程度、可能原因、处理建议、禁止操作、是否建议找师傅、参考价格和维修记录。

产品的核心差异化不是“AI 能回答维修问题”，而是：

- 结构化诊断：把用户自然语言转成故障分类、紧急等级、处理建议、价格参考和维修记录。
- 安全兜底：对燃气、漏电、冒烟、严重漏水等场景使用规则和模型双重判断，优先输出安全动作。
- 记录资产化：把一次性咨询沉淀为家庭维修档案，为复发追踪、费用统计和保养提醒提供基础。
- 闭环可扩展：后续可接入物业、房东、维修服务商和设备保养服务，不停留在泛化问答。

### 2.3 产品目标

MVP 阶段目标：

- 建立一条完整的家庭故障 AI 诊断链路。
- 覆盖更多高频家庭维修场景，MVP 首批覆盖 10 类问题。
- 支持文字、图片和语音输入，适配用户在移动端即时描述故障的习惯。
- 让用户在 3 分钟内完成一次初步诊断。
- 明确区分“可自行排查”和“必须联系专业人员”的场景。
- 控制单次诊断 AI 成本，避免留存越高亏损越大。
- 满足微信小程序、生成式 AI、内容安全和个人信息保护的上线合规要求。
- 沉淀用户维修记录，为后续家庭资产管理和维修服务转化做准备。
- 采用“移动端 H5 先上线、小程序后适配”的路线，优先验证产品链路，避免第一阶段被小程序审核和备案周期拖慢。

长期目标：

- 从单次维修诊断入口，延展为家庭维修记录、设备档案、保养提醒、房屋健康报告和维修服务交易平台。
- 形成“故障诊断 -> 维修建议 -> 维修记录 -> 设备档案 -> 保养提醒 -> 家庭资产管理”的产品闭环。

## 3. 目标用户

### 3.1 核心用户

| 用户类型 | 典型特征 | 核心需求 |
| --- | --- | --- |
| 自住房业主 | 对房屋长期维护负责，关注维修质量和复发问题 | 快速判断问题原因、风险和维修成本 |
| 租房用户 | 缺少维修经验，不清楚该找房东、物业还是师傅 | 明确处理路径和沟通依据 |
| 城市独居用户 | 遇到紧急故障时缺少现场协助 | 获得安全提示和可执行的下一步建议 |
| 新装修家庭 | 对渗漏、墙面、门窗、电路等问题敏感 | 判断是否涉及装修质量或保修 |
| 有老人或小孩的家庭 | 更关注用电、燃气、漏水等安全风险 | 快速识别高风险并避免错误操作 |
| 房东/出租房管理者 | 需要管理多套房的维修历史和费用 | 记录维修过程、费用和复发情况 |

### 3.2 种子用户

MVP 优先服务以下用户：

- 城市年轻家庭。
- 新房业主。
- 租房独居人群。
- 小规模出租房房东。

这些用户的共同特点是：遇到家庭维修问题时缺少经验，同时对价格透明、安全判断和处理路径有明确需求。

## 5. 竞品分析与差异化

### 5.1 对比对象

| 类型 | 代表产品 | 用户现有做法 | 局限 |
| --- | --- | --- | --- |
| 通用 AI 助手 | 豆包、Kimi、通义、文心、DeepSeek 等 | 用户直接问“马桶堵了怎么办” | 答案泛化，缺少结构化记录、价格库、维修闭环和安全规则兜底 |
| 维修服务平台 | 58 到家、啄木鸟、万师傅、京东服务+ 等 | 用户直接下单找师傅 | 更偏交易，用户在下单前仍不知道问题严重程度和报价是否合理 |
| 物业报修系统 | 小区物业 App/小程序 | 用户提交报修单 | 通常只服务某个小区，诊断能力弱，难覆盖租房、房东和非物业责任问题 |
| 搜索/社区内容 | 百度、小红书、知乎等 | 用户搜索经验贴 | 信息质量不稳定，缺少个性化追问和风险兜底 |

### 5.2 产品差异化

- 相对通用 AI：本产品强调标准化维修诊断流程、风险分级、价格参考、维修记录沉淀和可配置规则，不只生成自然语言回答。
- 相对维修平台：本产品先帮助用户判断问题、风险和合理价格，再决定是否找师傅，降低盲目下单和报价不透明。
- 相对物业报修：MVP 独立可用，不依赖小区系统；后续接入物业后，可把诊断结果转为报修单和责任边界判断。
- 相对搜索社区：通过多轮追问收集上下文，给出更贴近用户现场情况的结构化建议。

### 5.3 护城河假设

- 结构化维修知识库和价格库。
- 高风险场景规则库和人工评测集。
- 维修记录、复发记录和家庭设备档案。
- 后续与物业、房东和维修服务商的闭环数据。

## 6. 产品范围

### 6.1 MVP 覆盖场景

MVP 覆盖 10 类高频且适合展示 AI 诊断价值的家庭维修场景：

| 场景 | 示例问题 | MVP 覆盖重点 |
| --- | --- | --- |
| 漏水渗水 | 卫生间天花板滴水、墙面渗水、水龙头漏水 | 判断位置、持续性、是否影响电路、是否需物业或师傅 |
| 马桶/下水堵塞 | 马桶水下不去、地漏返水、厨房下水慢 | 判断堵塞程度、是否可自行尝试、是否需疏通 |
| 空调不制冷 | 有风但不冷、制冷慢、显示故障代码 | 判断滤网、模式、室外机、制冷剂、售后路径 |
| 电路跳闸 | 全屋跳闸、某一路跳闸、插座发黑 | 判断安全风险、触发高危提示、建议断电找电工 |
| 门锁故障 | 门锁打不开、指纹锁失灵、钥匙卡住 | 判断电池、机械故障、是否被困、是否需开锁服务 |
| 墙面发霉/墙皮脱落 | 墙角发霉、墙皮鼓包、墙面返潮 | 判断潮湿来源、通风情况、是否疑似渗漏 |
| 热水器故障 | 不出热水、忽冷忽热、显示故障码 | 判断电热水器/燃气热水器类型、风险提示、售后路径 |
| 油烟机/燃气灶问题 | 油烟机吸力弱、燃气灶打不着火、有燃气味 | 判断清洁、点火、电池、燃气高风险兜底 |
| 地漏反味/厨卫异味 | 地漏臭味、厨房下水反味、卫生间异味 | 判断水封、堵塞、管道反味和处理路径 |
| 门窗五金/纱窗损坏 | 窗户关不严、门窗变形、纱窗破损 | 判断配件损坏、密封问题、是否影响安全 |

### 6.2 MVP 不做范围

- 不做全品类维修平台。
- 不做正式维修订单履约系统。
- 不做师傅实时派单和支付结算。
- 不做复杂 IoT 设备接入。
- 不做最终诊断承诺，只提供初步判断和决策建议。
- 不做实时电话客服或人工坐席转接。
- 第 1 版不做微信小程序正式上线，不把小程序审核通过作为 MVP 验收前提。
- 第 1 版不强制接入图片/语音上传和登录，可先完成文字诊断、结果页和基础记录；图片/语音上传和基础登录进入第 2 版。

## 7. 用户主链路

### 7.1 标准诊断链路

1. 用户进入首页。
2. 用户在聊天界面中通过文字、语音或图片描述故障。
3. AI 识别故障类型和潜在风险。
4. AI 根据故障类型动态追问关键信息。
5. 用户补充文字、语音或图片。
6. AI 输出结构化诊断结果。
7. 用户查看紧急程度、处理建议、禁止操作和参考价格。
8. 用户选择保存为维修记录。
9. 用户登录或授权后，系统将匿名诊断记录合并到正式账号。
10. 系统在维修后通过微信订阅消息提醒用户回填实际维修方式、费用、服务商和是否解决；若用户不回填，记录仍保留 AI 诊断结果。

### 7.2 示例链路

用户输入：“我家卫生间天花板一直滴水。”

AI 追问：

- 是持续滴水还是偶尔渗水？
- 漏水位置是否靠近灯具、插座或电器？
- 是否能关闭附近电源？
- 是否方便上传照片？

AI 输出：

- 故障类型：卫生间漏水/渗水。
- 紧急程度：A 级，建议尽快处理。
- 可能原因：楼上防水层渗漏、吊顶内管道接口渗漏、公共管道问题。
- 安全提示：远离电源，不建议自行拆吊顶。
- 处理建议：拍照留存，联系物业确认是否涉及楼上或公共管道，必要时联系专业水电师傅。
- 参考价格：检测 50-150 元，局部维修 200-800 元，防水重做需现场评估。
- 下一步：保存维修记录。

## 8. 功能需求

> 本章为产品**终态功能规格**。各版本实际交付范围以第 9 章（功能优先级）和第 13 章（部署上线策略）为准。第 1 版（移动端 H5）仅交付文字诊断、多轮追问、故障分类、紧急程度、结构化结果页、价格参考和基础维修记录；语音、图片、登录和微信原生能力（如一键拨号、订阅消息）按版本路线分阶段接入。

### 8.1 首页与故障输入

#### 功能说明

首页直接采用聊天界面，不做营销落地页。用户打开产品后，应立即进入类似“AI 助手对话”的故障诊断空间，可以直接和 AI 通过文字、语音、图片沟通；常见故障入口作为聊天界面中的快捷建议，而不是独立表单。

#### 用户故事

作为家庭用户，我希望像和 AI 助手聊天一样描述问题，而不是先填写复杂表单；当我不方便打字时，也可以直接按住说话或上传照片，让系统帮我识别故障。

#### 输入

- 文字描述，选填，但文字、语音、图片至少提供一种。
- 语音输入，选填，MVP 支持最长 60 秒单条语音。
- 图片上传，选填，MVP 支持最多 3 张。
- 快捷故障入口，选填。

#### 处理逻辑

- 用户文字输入少于 5 个字时，提示补充故障现象。
- 用户发送语音后，先进行语音识别转文本，再进入故障识别和多轮追问流程。
- 语音识别结果需要展示给用户确认，允许用户一键编辑或重新录音。
- 图片上传后进入图片理解流程，识别漏水、发霉、烧黑、堵塞、故障代码、门锁状态等可视信息。
- 用户点击快捷入口后，在聊天中自动发送对应场景提示，并由 AI 开始追问。

#### 输出

- 创建一次聊天式诊断会话。
- 保存文字、语音识别文本、图片和快捷入口来源。
- 进入 AI 多轮问诊流程。

#### 边界

- 不要求用户一开始填写地址、房屋信息或联系方式。
- 不在首页直接给维修服务广告，避免破坏信任。
- 语音识别失败时，不阻断诊断，提示用户重新录音或改用文字描述。

### 8.2 AI 多轮追问

#### 功能说明

AI 根据故障类型动态生成追问，补齐判断所需的关键字段。多轮追问是产品核心亮点，目标是让 AI 像维修顾问一样逐步确认问题，而不是直接生成泛化答案。

#### 用户故事

作为缺少维修经验的用户，我希望系统主动问我关键问题，这样我不用知道应该提供哪些信息。

#### 场景追问字段

| 场景 | 必问字段 |
| --- | --- |
| 漏水渗水 | 漏水位置、持续时间、是否扩大、是否靠近电源、是否来自楼上、是否有照片 |
| 马桶/下水堵塞 | 堵塞位置、是否返水、是否完全不下水、是否尝试疏通、是否伴随异味 |
| 空调不制冷 | 使用年限、是否有风、是否清洗滤网、室外机是否运行、是否有故障代码 |
| 电路跳闸 | 全屋还是局部、使用哪个电器时发生、是否闻到焦味、插座是否发热或烧黑 |
| 门锁故障 | 指纹锁还是机械锁、是否人在门内/门外、是否电池没电、钥匙是否卡住 |
| 墙面发霉/墙皮脱落 | 位置、面积、是否靠近卫生间/外墙、是否返潮、是否近期漏水 |
| 热水器故障 | 热水器类型、是否有燃气味、是否显示故障码、水温变化、使用年限 |
| 油烟机/燃气灶问题 | 设备类型、是否有燃气味、是否能点火、是否近期清洁、是否有异响 |
| 地漏反味/厨卫异味 | 异味位置、持续时间、是否返水、是否长期不用、是否下水慢 |
| 门窗五金/纱窗损坏 | 门窗位置、是否关不严、是否变形、是否影响防盗/坠落安全 |

#### 处理逻辑

- 每次最多追问 3 个问题，避免用户负担过重。
- 每次诊断最多追问 3 轮；用户答非所问或信息仍不足时，用现有信息生成低置信度结果，并明确提示不确定原因。
- 如果命中高风险关键词，立即插入安全提示。
- 用户已经提供的信息不重复追问。
- 信息足够时停止追问并生成结果。

#### 输出

- 结构化字段补全结果。
- 诊断结果生成所需上下文。

#### 边界

- 不追问与诊断无关的隐私信息。
- 不诱导用户进行高风险维修操作。

### 8.3 故障分类

#### 功能说明

系统将用户自然语言和图片信息转为标准故障分类，供后续诊断、价格估算、记录沉淀和数据统计使用。

#### 一级分类

- 水路维修。
- 电路维修。
- 家电维修。
- 门锁门窗。
- 墙面地面。
- 厨卫设备。
- 日常保养。

#### MVP 二级分类

- 漏水渗水。
- 马桶/下水堵塞。
- 空调不制冷。
- 电路跳闸。
- 门锁故障。
- 墙面发霉/墙皮脱落。
- 热水器故障。
- 油烟机/燃气灶问题。
- 地漏反味/厨卫异味。
- 门窗五金/纱窗损坏。

#### 二级分类与一级分类映射

为避免分类器在归属上产生分歧，二级分类固定映射到唯一一级分类：

| 二级分类 | 一级分类 | 归属说明 |
| --- | --- | --- |
| 漏水渗水 | 水路维修 | 管路渗漏 |
| 马桶/下水堵塞 | 水路维修 | 排水管道堵塞 |
| 空调不制冷 | 家电维修 | 家电品类，涉及售后和质保 |
| 电路跳闸 | 电路维修 | 供电线路与配电 |
| 门锁故障 | 门锁门窗 | 门禁与五金 |
| 墙面发霉/墙皮脱落 | 墙面地面 | 墙地面饰面问题 |
| 热水器故障 | 家电维修 | 归家电品类（虽常装在厨卫），走售后与质保 |
| 油烟机/燃气灶问题 | 厨卫设备 | 厨卫设备与燃气接口 |
| 地漏反味/厨卫异味 | 厨卫设备 | 侧重器具与管道接口问题，区别于漏水的管路维修 |
| 门窗五金/纱窗损坏 | 门锁门窗 | 门窗与五金 |

#### 安全风险属性

安全风险不作为故障分类，而是作为横切属性由紧急程度模块承载。同一个故障可以同时属于“厨卫设备/油烟机/燃气灶问题”和“S 级安全风险”，避免分类器在故障类型和风险等级之间冲突。

#### 处理逻辑

- 大模型负责理解自然语言描述。
- 图片模型辅助识别可视故障。
- 分类结果需要输出置信度。
- 置信度较低时，优先通过追问补充信息，而不是直接输出结论。

#### 输出

```json
{
  "primary_category": "水路维修",
  "secondary_category": "漏水渗水",
  "confidence": 0.86,
  "evidence": ["用户描述天花板滴水", "图片显示吊顶边缘水痕"]
}
```

#### 边界

- 如果无法分类，输出“暂无法判断”，并提示用户补充描述或图片。
- 不把分类结果包装成最终维修结论。

### 8.4 紧急程度判断

#### 功能说明

系统根据故障类型、关键词、图片信息和用户补充信息，判断紧急程度，并给出明确处理建议。

#### 等级定义

| 等级 | 定义 | 示例 | 产品动作 |
| --- | --- | --- | --- |
| S 级 | 立即处理，存在人身或重大财产风险 | 燃气味、漏电、冒烟、插座火花、严重漏水影响电路、玻璃破裂伤人风险 | 强提示，优先安全指引，建议立即联系专业机构 |
| A 级 | 尽快维修，持续扩大或影响正常生活 | 持续漏水、频繁跳闸、门锁无法打开、严重堵塞 | 建议尽快联系物业、售后或专业师傅 |
| B 级 | 预约维修，不宜长期拖延 | 空调不制冷、水龙头漏水、地漏反味、家电异响 | 给排查步骤和预约维修建议 |
| C 级 | 可自行排查或保养 | 空调滤网脏、门锁电池低、水压短时波动 | 给安全的自查步骤 |

#### 高风险兜底规则

出现以下词或同义表达时，先召回高风险候选，再由语义判断确认是否为真实风险描述：

- 燃气、煤气、燃气味、刺鼻气味。
- 漏电、触电、麻手。
- 冒烟、火花、烧焦、插座烧黑。
- 严重漏水、漏到电器、灯具进水。
- 被困、门打不开、老人小孩被锁。
- 玻璃破裂、高空坠物。

#### 处理逻辑

- 大模型负责理解语义。
- 规则引擎负责高风险关键词召回和动作兜底。
- 关键词召回后，需要识别否定和假设表达，例如“没有燃气味”“担心是不是燃气泄漏”。
- 真实风险被确认时强制 S 级；否定或假设被明确排除（如用户确认“没有燃气味”）时可解除 S 级。
- 当无法确认风险是否真实存在时，**维持高风险提示并追问确认，不降级**，与第 18.6 节“规则引擎异常时高风险场景默认按更高风险处理”保持一致（安全场景宁可误报不可漏报）。
- 当大模型低估明确高风险描述时，高风险规则优先。

#### 输出

- 紧急等级。
- 风险原因。
- 当前最重要的安全动作。
- 是否建议立即联系专业人员。

#### 边界

- 对高风险场景，不输出具体维修教学。
- 不建议用户自行拆电路、燃气、承重结构或高空设备。

### 8.5 诊断结果页

#### 功能说明

诊断结果页是一次 AI 诊断的核心交付物，必须结构化展示，不允许只输出一段长文本。

#### 页面模块

| 模块 | 内容 |
| --- | --- |
| 故障类型 | 一级分类、二级分类、置信度 |
| 紧急程度 | S/A/B/C 等级、颜色标识、原因 |
| 可能原因 | 2-4 个可能原因，按概率或常见程度排序 |
| 建议处理方式 | 自查、联系物业、联系售后、找专业师傅 |
| 可自行排查步骤 | 只展示低风险步骤 |
| 禁止操作提醒 | 高风险场景必须展示 |
| 是否建议找师傅 | 是/否/视情况，并说明理由 |
| 价格区间 | 常见市场参考区间和影响因素 |
| 维修前准备 | 拍照、断电、关闭阀门、保留票据等 |
| 保存记录 | 一键保存到维修记录 |

#### 输出要求

- 先给结论，再给原因。
- 安全提示优先级高于维修建议。
- 价格必须标注为参考区间。
- 不确定时明确说明不确定，并建议补充信息或线下检查。

### 8.6 价格区间参考

#### 功能说明

为用户提供常见维修服务价格区间，帮助用户降低对报价不透明的焦虑。

#### MVP 价格库

价格库需要覆盖 MVP 10 类场景，按城市能级维护，不追求精确报价。MVP 使用“一线城市/其他城市”两档，无法可靠估价的场景展示价格影响因素和量级。

| 场景 | 一线城市参考 | 其他城市参考 | 说明 |
| --- | --- | --- | --- |
| 漏水渗水 | 检测 80-200 元，局部维修 300-1000 元 | 检测 50-150 元，局部维修 200-800 元 | 防水重做、墙体开凿需现场评估 |
| 马桶/下水堵塞 | 150-400 元 | 80-300 元 | 严重堵塞、夜间上门可能加价 |
| 空调不制冷 | 清洗 150-350 元，加氟 200-500 元 | 清洗 100-300 元，加氟 150-400 元 | 压缩机、电路板维修需另估 |
| 电路跳闸 | 150-600 元 | 100-500 元 | 涉及配电箱、线路改造需现场评估 |
| 门锁故障 | 150-600 元 | 100-500 元 | 智能锁配件和开锁服务差异较大 |
| 墙面发霉/墙皮脱落 | 小面积处理 200-800 元 | 小面积处理 150-600 元 | 若源头是渗漏，需先处理漏水 |
| 热水器故障 | 检修 150-500 元 | 检修 100-400 元 | 燃气类优先联系品牌售后或燃气相关服务 |
| 油烟机/燃气灶问题 | 清洗/检修 150-500 元 | 清洗/检修 100-400 元 | 有燃气味时不进入报价优先安全处理 |
| 地漏反味/厨卫异味 | 100-400 元 | 80-300 元 | 可能涉及水封、管道、堵塞或返味 |
| 门窗五金/纱窗损坏 | 150-600 元 | 100-500 元 | 高层外窗风险场景需专业人员处理 |

#### 价格库字段

| 字段 | 说明 |
| --- | --- |
| scene_code | 场景编码 |
| city_tier | 城市能级，一线/其他 |
| min_price | 参考最低价 |
| max_price | 参考最高价 |
| price_unit | 次/件/平方米/现场评估 |
| source_note | 来源说明 |
| version | 价格版本 |
| updated_at | 更新时间 |

#### 处理逻辑

- 根据故障分类匹配价格区间。
- 如果需要现场评估，明确提示无法线上给出精确报价。
- 支持展示影响价格的因素，如城市、上门费、材料费、维修难度、是否夜间上门。

#### 固定提示

以下为常见市场参考区间，实际价格取决于城市、材料、上门费、现场情况和服务商报价。请以维修前确认的报价为准。

#### 边界

- 不承诺最终价格。
- 不替用户判断服务商报价是否违法或欺诈。
- 不在 MVP 中做支付或担保交易。

### 8.7 风险与安全提示

#### 功能说明

针对燃气、漏电、冒烟、严重漏水、门锁被困等场景，系统必须给出明确安全提示，优先保障用户安全。

#### 典型提示规则

| 场景 | 必须提示 |
| --- | --- |
| 燃气味 | 不要开关电器，立即开窗通风，关闭阀门，撤离现场，联系当地燃气公司；情况紧急时拨打 119 |
| 插座烧黑/冒烟 | 不要继续使用，关闭对应空气开关，远离现场，联系专业电工；冒烟或起火时拨打 119 |
| 严重漏水 | 远离电源，关闭总阀或相关阀门，拍照取证，联系物业或专业人员；触电或受伤时拨打 120 |
| 门锁无法打开 | 不要暴力拆锁，确认是否有人被困，联系正规开锁服务；老人小孩被困或有危险时拨打 110/119 |
| 玻璃破裂 | 远离破损区域，避免自行高空处理，联系专业人员；有伤情时拨打 120 |

#### 一键拨打

不同端的拨号实现方式不同：移动端 H5 使用 `tel:` 协议链接拨号，微信小程序使用 `makePhoneCall` 原生能力。号码配置优先级：

1. 用户手动保存的物业电话、紧急联系人、服务商电话。
2. 城市或小区配置的燃气公司、物业服务热线。
3. 全国紧急电话：119、120、110。

#### 边界

- AI 不提供高风险维修操作教学。
- AI 不替代燃气公司、电工、消防、物业或专业维修人员。
- 高风险提示必须在结果页顶部展示。

### 8.8 维修记录

#### 功能说明

用户可将每次诊断保存为维修记录，并在维修后补充实际处理结果，形成家庭维修档案。

#### 字段设计

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| 记录 ID | string | 是 | 系统生成 |
| 创建时间 | datetime | 是 | 诊断时间 |
| 房屋/房间 | string | 是 | 保存记录时必选，提供快捷选项兜底 |
| 故障类型 | enum | 是 | 诊断分类 |
| 故障描述 | text | 是 | 用户原始描述 |
| 图片 | array | 否 | 上传图片 |
| AI 诊断结果 | json | 是 | 类型、等级、建议、价格 |
| 实际维修方式 | text | 否 | 用户后续补充 |
| 维修费用 | number | 否 | 实际支出 |
| 服务商/师傅 | string | 否 | 用户自行记录 |
| 是否解决 | boolean | 否 | 是否修好 |
| 是否复发 | boolean | 否 | 是否再次出现 |

#### 页面能力

- 查看历史维修记录列表。
- 按故障类型、时间、房间筛选。
- 查看单条记录详情。
- 补充实际维修结果。
- 标记是否复发。
- 保存记录时必选房间，提供厨房、卫生间、客厅、卧室、阳台、入户门、“其他/不确定”等快捷选项兜底，保证按房间筛选和复发追踪的数据质量。
- 支持设置维修后 1 天/3 天微信订阅消息提醒，提示用户回填实际费用和是否解决。

#### 后续价值

- 生成家庭维修档案。
- 统计年度维修支出。
- 识别高频故障。
- 生成设备保养提醒。

### 8.9 用户反馈

#### 功能说明

用户在查看诊断结果后，可以反馈诊断是否有帮助，用于评估 AI 效果。

#### 反馈项

- 诊断是否有帮助：有帮助/一般/没有帮助。
- 故障类型是否准确：准确/不准确。
- 紧急程度是否准确：准确/偏高/偏低。
- 是否已联系维修服务：是/否。
- 可选文本反馈。

#### 数据用途

- 优化故障分类。
- 优化多轮追问。
- 评估高风险规则召回。
- 建立后续人工修正样本。

## 10. 界面设计要求

### 10.1 设计原则

- 第一屏直接进入 AI 聊天界面，不做营销式落地页。
- 使用类似“蚂蚁阿福”这类 AI 助手的对话式界面，让用户自然发起咨询、补充图片和语音。
- 诊断结果必须结构化、可扫描、可保存。
- 高风险信息必须强提示，不被普通建议淹没。
- 避免使用过多专业术语，优先使用家庭用户能理解的表达。
- 微信小程序和移动端优先，兼容 Web/H5。
- 适老化优先：关键文字不低于 16px，主要按钮高度不低于 44px，高风险提示使用高对比度样式，语音交互允许重录和纠错。

### 10.2 信息架构

MVP 页面：

| 页面 | 核心内容 |
| --- | --- |
| 首页/诊断对话页 | AI 聊天窗口、文字输入、按住说话、图片上传、快捷故障建议、安全提示 |
| 诊断结果页 | 类型、紧急等级、可能原因、建议、价格、保存记录 |
| 维修记录页 | 历史记录列表、筛选、详情 |
| 我的家庭页 | 房屋、房间、设备基础信息 |
| 反馈页 | 诊断准确性和有用性反馈 |

### 10.3 首页设计要求

- 首页即聊天页，默认展示 AI 欢迎语和 3-5 个常见问题建议，不出现复杂表单。
- 底部固定输入栏，包含文字输入、语音按钮、图片上传按钮和发送按钮。
- 语音按钮采用移动端常见的“按住说话/松开发送/上滑取消”交互。
- 图片上传入口与输入栏相邻，支持拍照和从相册选择。
- 常见故障快捷入口使用建议卡片或横向标签：
  - 漏水。
  - 马桶堵了。
  - 空调不冷。
  - 跳闸。
  - 门锁打不开。
- 用户点击快捷入口后，在聊天区自动生成一条用户问题，并由 AI 继续追问。
- 不要求登录即可开始一次诊断；保存记录时再引导登录或本地保存。

### 10.4 诊断对话页设计要求

- 使用左右气泡展示用户与 AI 的对话。
- 用户消息支持文字、语音转写文本、图片缩略图和快捷入口消息。
- AI 消息支持追问、提示卡片、风险警示和结果摘要。
- AI 每轮追问不超过 3 个问题。
- 用户可随时补充图片或语音。
- 命中高风险时，在对话上方展示固定安全提示条。
- 提供“信息差不多了，生成诊断结果”按钮。
- 语音识别文本在发送后以消息形式展示，用户可长按修改或重新发送。

### 10.5 诊断结果页设计要求

- 顶部展示故障类型和紧急等级。
- S/A 级使用醒目的风险视觉样式。
- “现在应该做什么”模块优先于“可能原因”模块。
- 禁止操作使用独立警示块。
- 价格区间模块需要带免责声明。
- 底部提供主操作：
  - 保存为维修记录。
  - 补充信息重新诊断。
  - 复制维修沟通清单。

### 10.6 维修记录页设计要求

- 列表项展示故障类型、房间、紧急等级、时间、是否解决、实际费用。
- 详情页展示完整 AI 诊断、图片、维修结果和复发标记。
- 支持按时间倒序排列。
- 支持筛选未解决、已解决、复发问题。

## 13. 部署上线策略

### 13.1 总体原则

产品采用“先 H5 快速上线，再逐步增强微信生态能力”的路线。第一阶段不以微信小程序正式审核通过作为上线目标，而是先让用户可以通过手机浏览器或微信内置浏览器打开链接，完成核心诊断链路。

### 13.2 分阶段上线方案

| 阶段 | 上线形态 | 核心能力 | 目标 |
| --- | --- | --- | --- |
| 第 1 版 | 移动端 H5 + FastAPI + PostgreSQL | 文字聊天诊断、结构化结果页、基础维修记录、RAG/规则/LLM 调用 | 1-3 天内获得可手机访问的 MVP，验证核心链路 |
| 第 2 版 | H5 正式部署增强 | 域名、HTTPS、图片上传、语音上传/识别、基础登录、内容安全审核 | 形成可对外测试的完整体验 |
| 第 3 版 | 微信小程序体验版 | 复用 H5 核心能力，适配小程序录音、拍照、登录、一键拨号 | 用于内部测试、演示和小范围体验 |
| 第 4 版 | 微信小程序正式版 | 小程序审核、隐私接口说明、内容安全、合规材料、用户协议、线上运营配置 | 面向公众正式发布 |

### 13.3 第 1 版范围

第 1 版优先保证能上线、能访问、能完成一次诊断闭环：

- 前端采用移动端 H5 页面，优先适配微信内置浏览器和手机浏览器。
- 后端采用 FastAPI。
- 数据库采用 PostgreSQL。
- 输入方式优先支持文字，图片/语音可先保留入口或提示“即将支持”。
- 支持 AI 多轮追问、故障分类、紧急程度判断、结构化结果页。
- 支持基础维修记录保存，可先使用游客会话或临时用户标识。
- 使用临时部署地址也可验收，域名和 HTTPS 进入第 2 版。
- 第 1 版不强制通过第 21.4 节的评测阈值验收，评测集和阈值从第 2 版（对外测试）起强制。
- 第 1 版 H5 端使用 localStorage 生成的匿名设备标识临时归属会话；该标识不可靠（清缓存或换浏览器即丢失），匿名会话合并为“尽力而为”，免费额度限制在 H5 端可被绕过。正式可靠的账号合并与额度管控在小程序阶段（openid）生效。

### 13.4 第 2 版范围

第 2 版补齐对外测试所需能力：

- 绑定正式域名并启用 HTTPS。
- 接入图片上传和图片安全审核。
- 接入语音上传、ASR 语音识别和转写文本安全审核。
- 接入基础登录，优先支持手机号或验证码登录，后续再接微信授权。
- 完善隐私政策、用户协议、免责声明和内容安全提示。
- 支持维修记录和登录账号绑定。

### 13.5 第 3 版范围

第 3 版适配微信小程序体验版：

- 使用 uni-app 复用 H5 业务页面。
- 适配微信小程序录音、拍照、文件上传、一键拨号能力。
- 使用微信 `openid` 做匿名用户标识。
- 小程序体验版用于演示、测试、面试或小范围真实用户反馈。
- 不把正式审核通过作为该阶段验收标准。

### 13.6 第 4 版范围

第 4 版再推进微信小程序正式上线：

- 完成小程序类目、主体、域名、隐私接口和代码审核。
- 完成生成式 AI、内容安全、隐私政策、用户协议和投诉反馈机制。
- 接入微信订阅消息，用于维修后回填提醒。
- 完成线上监控、成本统计、内容安全日志和合规留痕。

## 14. 技术栈建议

### 14.1 前端

- 移动端 H5：第 1 版首发形态，优先满足手机浏览器和微信内置浏览器访问。
- uni-app：作为跨端框架，一套 Vue 技术栈先发布 H5，后续适配微信小程序。
- 微信小程序：第 3 版开始适配体验版，第 4 版推进正式上线。
- Vue 3 + TypeScript：适合 uni-app/Vue 技术路线，提升工程可维护性。
- Pinia：管理诊断会话、用户状态、维修记录状态。
- Tailwind CSS、UnoCSS 或 NutUI/TDesign Mobile：快速构建设计一致的移动端界面。
- Web/H5：作为兼容端，复用核心页面与接口能力，不作为 MVP 第一优先级。

### 14.2 后端

- FastAPI：适合 AI 产品后端，接口开发快，Python 生态完整。
- PostgreSQL：第 1 版即采用，支持结构化记录、账号合并、成本日志和后续分析。
- SQLite：仅用于本地开发或 Demo，不作为上线数据库。
- SQLAlchemy 或 SQLModel：管理数据模型。
- Redis：后续用于会话缓存、限流和异步任务状态。

### 14.3 AI 与检索

- LLM：负责对话理解、追问生成、诊断结果生成。MVP 优先选择已完成国内合规备案、低延迟、中文能力稳定的国产模型服务，通过 LLM Adapter 接入，候选包括通义千问、文心、GLM、DeepSeek 等。
- ASR 语音识别：负责将用户语音转为文本，建议优先使用微信小程序录音能力 + 云端语音识别服务。
- 多模态模型：负责图片故障识别。
- RAG：连接维修知识库、安全规范、价格库和故障案例。
- 向量库：
  - MVP：优先使用 pgvector 或轻量本地向量库。
  - 后续：数据量增长后接入 Milvus。
- 规则引擎：用于高风险兜底、紧急等级提升、价格边界控制。

### 14.4 文件与存储

- MVP 图片和语音文件可先使用本地文件存储抽象。
- 后续兼容 OSS/S3。
- 图片需要保存原图地址、压缩图地址、上传时间、关联诊断 ID。
- 语音需要保存原始文件地址、识别文本、识别置信度、时长和关联诊断 ID。

### 14.5 推荐架构

```text
移动 H5 / 微信小程序
  -> FastAPI API
    -> Diagnosis Service
    -> Rule Engine
    -> ASR Adapter
    -> LLM Adapter
    -> RAG Retrieval
    -> Price Reference Service
    -> Record Service
  -> PostgreSQL（SQLite 仅用于本地开发，不上线）
  -> Vector Store
  -> Object Storage
```

## 15. 合规与法务

### 15.1 合规范围

本产品属于“微信小程序 + 面向公众的 AI 生成内容 + 用户上传图片/语音 + 涉及燃气/用电安全建议”的高敏组合。公开上线前必须完成合规评估，不能只按普通工具小程序处理。

### 15.2 生成式 AI 与算法合规

- 模型选型优先选择已完成国内合规备案、服务稳定、支持企业调用的国产模型服务。
- 若以自有服务面向公众提供生成式 AI 能力，需要按《生成式人工智能服务管理暂行办法》等要求准备备案、内容安全、投诉举报、个人信息保护等材料。
- 后端必须保留模型供应商、模型版本、Prompt 版本、知识库版本和诊断结果版本，用于问题追溯。
- 若模型供应商提供合规托管能力，产品仍需完成自身应用层的小程序审核、用户协议、隐私政策、内容安全和责任声明。

### 15.3 内容安全审核

| 内容类型 | 审核要求 |
| --- | --- |
| 用户文本 | 进入 AI 诊断前做涉政、涉黄、涉暴、辱骂、违法等内容检测 |
| 用户图片 | 上传后先做图片内容安全检测，再进入图片理解 |
| 用户语音 | 先转写文本，再对转写文本做内容安全检测；必要时保留音频审核接口 |
| AI 输出 | 输出前做安全审核，禁止生成高风险维修教学、违法内容和不当承诺 |

内容安全失败时，不进入诊断流程，提示用户重新描述家庭维修问题。

### 15.4 实名与账号

- 匿名会话归属标识按端区分：微信小程序使用 `openid`；移动端 H5 使用 localStorage 生成的匿名设备标识，该标识不可靠，仅用于临时归属。
- H5 阶段的匿名会话合并为“尽力而为”：清缓存、换浏览器或换设备会导致标识丢失，历史诊断记录可能无法合并；正式可靠的账号合并在小程序阶段（openid）生效。
- 用户保存维修记录、使用历史记录、设置提醒、绑定房屋或设备时，引导登录或手机号绑定。
- 若所接入的生成式 AI 服务或监管要求需要实名，登录链路需要支持手机号实名或微信实名能力。
- 匿名诊断会话在登录后合并到正式账号，合并前需要提示用户确认。

### 15.5 用户协议与责任声明

用户协议和诊断结果页必须明确：

- AI 只提供初步判断和决策辅助，不构成最终维修结论。
- 涉及燃气、漏电、火灾、结构安全、人身伤害风险时，应立即联系专业机构或紧急电话。
- 平台不对用户自行执行高风险操作导致的损失承担超出法律规定的责任。
- 价格区间仅供参考，不构成报价承诺。
- 用户上传的图片、语音和文本仅用于故障诊断、记录保存、服务改进和合规审计，使用范围需在隐私政策中说明。

### 15.6 合规上线门槛

公开上线前必须完成：

- 小程序类目、服务内容和隐私接口审核。
- 用户协议、隐私政策、儿童/老人友好提示。
- 文本、图片、语音、AI 输出内容安全链路。
- 生成式 AI 服务和模型供应商合规材料确认。
- 投诉举报、删除记录、撤回授权、客服反馈入口。

- 《生成式人工智能服务管理暂行办法》：https://www.cac.gov.cn/2023-07/13/c_1690898327029107.htm
- 《互联网信息服务深度合成管理规定》：https://www.cac.gov.cn/2022-12/11/c_1672221949354811.htm

## 16. 成本模型与商业约束

### 16.1 单次诊断成本构成

一次完整诊断可能包含：

- ASR 语音识别。
- 图片多模态识别。
- 1-3 轮 LLM 追问。
- RAG 检索。
- 结构化结果生成。
- 内容安全审核。
- 图片和语音存储。

### 16.2 MVP 成本上限

MVP 目标：单次完整诊断 AI 与内容安全成本控制在 1 元以内；普通文本诊断控制在 0.3 元以内。若实际供应商报价无法满足，需要减少免费额度、压缩追问轮数、降低图片识别频率或更换模型。

成本上限基于以下量级拆解（按国产模型当前公开报价估算，仅用于设定基线，上线后以 `cost_logs` 实际值为准并校正）：

| 环节 | 单次诊断调用规模 | 估算成本区间 |
| --- | --- | --- |
| LLM 对话（分类 + 最多 3 轮追问 + 结构化生成，含上下文累积） | 约 6-12k token | 0.05-0.25 元 |
| 内容安全审核（用户输入 + AI 输出） | 2 次 | 0.005-0.03 元 |
| ASR 语音识别（仅语音输入时） | 0-1 次，≤60 秒 | 0-0.06 元 |
| 图片多模态识别（仅图片输入时） | 0-1 次 | 0-0.25 元 |
| RAG 检索（向量召回 + 重排） | 1-3 次 | 0.005-0.03 元 |

按此拆解：纯文本诊断约 0.06-0.31 元，需通过结果缓存、低风险走规则模板压到 0.3 元以内；含图片和语音的完整诊断约 0.06-0.62 元，处于 1 元目标内。**多模态是成本主要变量**，必须控制重复识别和免费诊断中多模态的触发频率。

### 16.3 成本控制策略

- 免费用户每日最多 3 次完整诊断。
- 每次诊断最多 3 轮追问，每轮最多 3 问。
- 同一会话内复用分类、风险判断和 RAG 检索结果。
- 图片仅在用户上传时调用多模态识别，不做重复识别。
- 低风险、常见问题优先使用规则和知识库模板生成，减少大模型调用。
- 价格库、风险规则、追问模板配置化，避免每次都由大模型生成。
- 对 ASR、图片识别、LLM、RAG、内容安全分别记录调用次数和成本。

### 16.4 商业化约束

MVP 不以维修交易收入覆盖成本为前提，必须通过免费次数、模型降级和缓存策略控制亏损。进入 V2.0 维修服务转化前，重点验证诊断完成率、结果满意度、记录保存率和服务点击意向。

## 18. 非功能性需求

### 18.1 性能

- 微信小程序首屏进入聊天页：移动网络下不超过 3 秒。
- Web/H5 首屏加载时间：移动网络下不超过 3 秒。
- 文本诊断首次响应：正常情况下不超过 5 秒。
- 语音识别响应：60 秒以内语音正常情况下不超过 8 秒返回转写结果。
- 图片上传单张大小：MVP 限制 10MB 以内。
- 图片诊断响应：正常情况下不超过 15 秒。
- 诊断结果页渲染：不超过 2 秒。

### 18.2 可用性

- 核心诊断链路优先适配微信小程序和移动端。
- Web/H5 保持核心链路可用，允许部分微信原生能力降级。
- 用户可以不登录完成一次诊断。
- 保存维修记录时再引导登录或本地保存。
- 匿名诊断按端使用归属标识：小程序用微信 `openid`，H5 用 localStorage 设备标识（不可靠）。登录或绑定手机号后支持一次性合并历史诊断和维修记录；H5 端因标识不可靠，合并为“尽力而为”。
- AI 失败时提供重试和人工化提示，不显示技术错误栈。

### 18.3 安全

- 上传图片需要校验文件类型和大小。
- 上传语音需要校验文件类型、时长和大小。
- 文本、图片、语音转写文本和 AI 输出都必须接入内容安全审核。
- 用户图片和维修记录默认仅用户本人可见。
- 不在日志中记录完整隐私文本、图片原始内容和语音原始内容。
- API 需要基础限流，防止恶意调用大模型。
- 高风险场景必须强制展示安全提示。

### 18.4 隐私

- 明确告知用户文字、图片和语音将用于故障诊断。
- 未经用户同意，不将个人维修记录用于公开展示。
- 支持用户删除诊断记录、图片和语音文件。
- 房屋地址、联系方式等敏感信息不作为 MVP 必填项。

### 18.5 成本与调用限制

- 单次完整诊断 AI 与内容安全成本目标不超过 1 元。
- 普通文本诊断成本目标不超过 0.3 元。
- 免费用户每日最多 3 次完整诊断。
- 每次诊断最多 3 轮追问，每轮最多 3 个问题。
- 成本超阈值时优先降级为规则模板、知识库回答或提示用户补充更明确的信息。

### 18.6 稳定性

- 大模型调用失败时，返回可理解的错误提示。
- RAG 检索失败时，可使用基础规则和通用知识降级输出。
- 价格库缺失时，不生成虚假价格，提示“暂无参考价格”。
- 规则引擎异常时，高风险场景默认按更高风险处理。

### 18.7 可维护性

- 故障分类、风险规则、价格区间、追问模板应配置化。
- AI Prompt 与业务代码分离。
- 诊断结果使用结构化 JSON 保存，便于后续分析。
- 知识库需要版本字段，便于后续更新和追溯。
- 价格库、安全规则、Prompt、知识库均需要版本号和更新时间。

## 19. 风控与产品边界

### 19.1 必须明确的边界

- AI 只提供初步判断，不承诺最终诊断。
- AI 不替代专业维修人员、物业、燃气公司、消防或电力人员。
- AI 不提供高风险维修操作教学。
- 涉及燃气、漏电、火灾、结构安全时，强制提示联系专业机构。
- 价格只做参考，不做最终报价。
- 图片识别不确定时，必须提示用户补充信息或线下检查。

### 19.2 禁止输出内容

- 指导用户自行拆燃气管道。
- 指导用户带电维修插座、开关、配电箱。
- 指导用户自行高空处理外墙、窗户、空调外机。
- 对结构安全给出确定性结论。
- 对服务商报价做无证据的欺诈判断。

### 19.3 高风险兜底文案

当前问题可能涉及人身或财产安全。AI 只能提供初步判断，不能替代专业人员现场检查。请优先确保人身安全，并联系物业、燃气公司、电工、消防或专业维修人员处理。

## 21. 验收标准

### 21.1 文档验收

- PRD 覆盖产品概述、目标用户、功能列表、优先级、界面要求、技术栈、非功能需求。
- 每个 MVP 核心功能包含用户故事、输入、处理逻辑、输出和边界。
- 明确 MVP 与后续版本范围，不把完整维修平台塞进 MVP。
- 明确 AI 的能力边界和高风险风控规则。
- 明确合规、内容安全、账号合并、成本上限和评测集要求。

### 21.2 产品链路验收

以下 10 个样例必须能在 PRD 设计中形成完整链路：

| 输入 | 预期链路 |
| --- | --- |
| 卫生间天花板滴水 | 追问位置/持续性/电源风险 -> A 级或 S 级判断 -> 漏水建议 -> 价格参考 -> 保存记录 |
| 空调开半小时不制冷 | 追问年限/滤网/室外机/故障码 -> B 级判断 -> 自查和售后建议 -> 价格参考 |
| 插座发黑还能用吗 | 命中高风险 -> S 级判断 -> 禁止继续使用 -> 断电并联系电工 |
| 马桶堵了水下不去 | 追问是否返水/堵塞程度 -> A/B 级判断 -> 可尝试低风险排查或联系疏通 |
| 门锁打不开 | 追问人在门内外/电池/钥匙状态 -> 判断是否紧急 -> 正规开锁或更换电池建议 |
| 墙角发霉越来越大 | 追问位置/面积/是否靠外墙或卫生间 -> 判断返潮或渗漏可能 -> 通风除湿和排查建议 |
| 热水器不出热水 | 追问类型/故障码/是否有燃气味 -> 风险判断 -> 售后或专业维修建议 |
| 燃气灶打不着火还有味道 | 命中燃气高风险 -> S 级提示 -> 不开关电器、通风、关阀、联系燃气公司 |
| 地漏一直反味 | 追问是否返水/是否长期不用 -> B/C 级判断 -> 水封、清洁、疏通建议 |
| 窗户关不严有松动 | 追问楼层/是否有坠落风险/五金状态 -> 风险判断 -> 五金维修或专业处理建议 |

### 21.3 技术验收

- 技术方案支持文本、语音、图片、多轮问诊、结构化结果和维修记录。
- 高风险规则不完全依赖大模型。
- 价格区间可配置，不写死在生成文案中。
- 诊断结果可结构化保存，支持后续数据分析。
- 匿名诊断可通过 `openid` 归属，登录后可合并到正式账号。
- 图片、语音、文本和 AI 输出均接入内容安全审核。
- AI 调用链路记录模型版本、Prompt 版本、知识库版本和成本日志。

### 21.4 AI 评测验收

- 建立不少于 200 条人工标注黄金集，覆盖 MVP 10 类场景。
- 每条样本标注故障一级分类、二级分类、紧急等级、是否高风险、推荐处理路径。
- 高风险样本至少 50 条，覆盖燃气、漏电、冒烟、严重漏水、门锁被困、玻璃破裂。
- 评测验收按版本分级：
  - 第 1 版（内部演示）：不强制过评测阈值，但需跑通评测流程并产出基线数据。
  - 第 2 版（对外测试）：故障分类准确率、紧急度判断准确率和高风险召回率达到产品设定阈值，作为对外可用的前提。
  - 第 4 版（正式上线）：阈值达标 + 内容安全审核通过率达标 + 用户协议/隐私政策合规通过。

## 23. 附录：MVP Prompt 输出要求

AI 生成诊断结果时，必须遵循以下要求：

- 必须输出故障类型。
- 必须输出紧急等级。
- 必须输出安全提示。
- 必须输出可能原因。
- 必须输出建议处理方式。
- 必须输出是否建议找师傅。
- 能匹配价格库时输出参考价格。
- 不确定时必须说明不确定原因。
- 高风险场景不得输出具体维修操作步骤。

示例输出结构：

```text
初步判断：疑似卫生间漏水/渗水。
紧急程度：A 级，建议尽快处理。
为什么：用户描述天花板持续滴水，可能涉及楼上防水层或吊顶内管道。
现在先做：远离附近电源，拍照记录漏水位置，联系物业确认是否涉及楼上或公共管道。
不要做：不要自行拆吊顶，不要在漏水附近继续使用电器。
是否找师傅：建议联系物业或专业水电师傅现场排查。
参考价格：检测 50-150 元，局部维修 200-800 元，防水重做需现场评估。
价格说明：实际价格取决于城市、材料、上门费和现场情况。
```
